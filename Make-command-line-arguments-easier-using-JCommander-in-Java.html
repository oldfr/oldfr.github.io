Make command-line arguments easier using JCommander in Java

JCommander is an annotation-based library for parsing command-line parameters. Using this we can make command-line arguments more manageable.

It allows us to provide all arguments as fields of a class, each annotated with corresponding argument details.

To begin with, we need the below dependency in pom.xml file:

    <dependency>
      <groupId>com.beust</groupId>
      <artifactId>jcommander</artifactId>
      <version>1.82</version>
    </dependency>

Let's create a class that has all command-line arguments:

Arguments class:

package org.example;

import com.beust.jcommander.Parameter;

import java.util.List;

public class Arguments {

    @Parameter(names = "--help", help = true )
    public boolean help = false;

    @Parameter(names = {"--name", "-n"}, required = true, description = "name of the student")
    public String name;

    @Parameter(names = {"--age", "-a"}, required = true, description = "age of the student")
    public int age;

    @Parameter(names = "--subjects",required = true, description = "Subjects student opted for")
    public List<String> subjectList;

    @Parameter(names = {"-password","-p"}, description = "Connection password", password = true, required = true, echoInput = false)
    public String studentPassword;

    @Parameter(names = "--petName", description = "pet name of the student")
    public String petName;

    @Parameter(names = "--isSpeciallyAbled", description = "Flag if student is specially abled")
    public Boolean isSpeciallyAbled;

}

In this class, we have six fields each annotated @Parameter which take details of an argument.

Below are some of the options @Parameter takes:

  • names —The name of arguments, taking an array of strings. We can provide many names to any argument until till does not lead to ambiguity.
  • help — If we want to provide an option that prints the details of involved arguments, we can make this flag true.
  • required — If any argument is mandatory, we will make it as required=true
  • description — It is an optional argument to provide detail of any particular argument.
  • password — If any field is sensitive and you do not wish to display it as plaintext on the console, we make this field true.
  • echoInput — If an argument is sensitive and you wish to display it on the console, then make this field true.

Note:

  • If any argument is a list, we need to provide them as comma-separated values.
  • If any argument is true or false (boolean datatype), we just provide the argument’s name. It will be interpreted as true. If we want to provide any default value, initialize the field:
  @Parameter(names = "--isSpeciallyAbled", description = "Flag if student is specially abled")
    public Boolean isSpeciallyAbled = false;

Main class:

package org.example;

import com.beust.jcommander.JCommander;

import java.util.List;
import java.util.Objects;


public class App 
{
    public static void main( String[] args )
    {
        Arguments arguments = new Arguments();
        JCommander jCommander = JCommander.newBuilder().addObject(arguments).build();
        jCommander.parse(args);

        if(arguments.help) {
            jCommander.usage();
            return;
        }

        String studentName = arguments.name;
        int studentAge = arguments.age;
        List<String> subjectsOpted = arguments.subjectList;
        String studentPetName = arguments.petName;
        String studentPassword = arguments.studentPassword;
        Boolean studentIsSpeciallyAbled = arguments.isSpeciallyAbled;
        System.out.println("Student Name: "+studentName);
        System.out.println("Student age: "+studentAge);
        System.out.println("Student opted for subjects: "+String.join(", ",subjectsOpted));
        System.out.println("Student password: "+studentPassword);
        System.out.println("Student is specially abled:"+studentIsSpeciallyAbled);
        if(Objects.nonNull(studentPetName))
            System.out.println("Student pet name: "+studentPetName);
    }
}

Here, we will build JCommander with the object of Argument class and parse it.

We can include all arguments in Run configurations in IntelliJ or we can generate a jar and invoke from the terminal.

Custom Type Parameters:

In addition to these features, we have the flexibility to bind parameters to custom types. We need to use IStringConverter interface and override convert() method to customize the conversion.

For example, we need to provide a hostname and port as an argument separate by a colon (:). We can create a holder class:

package org.example.model;

public class HostPort {

    final String host;
    final Integer port;

    public HostPort(String host, Integer port) {
        this.host = host;
        this.port = port;
    }

    @Override
    public String toString() {
        return "HostPort{" +
                "host='" + host + '\'' +
                ", port=" + port +
                '}';
    }
}

Create a converter class that implements IStringConverter interface:

package org.example.converter;

import com.beust.jcommander.IStringConverter;
import org.example.model.HostPort;

public class HostPortConverter implements IStringConverter<HostPort> {
    @Override
    public HostPort convert(String value) {
        String[] s = value.split(":");
        return new HostPort(s[0], Integer.parseInt(s[1]));
    }
}

In Argument class, the field needs a converter option:

    @Parameter(names = "--hostaddr", description = "hostname and port", converter = HostPortConverter.class)
    public HostPort hostPort;

Execution command:

java -jar jcommander-overview-1.0-SNAPSHOT-jar-with-dependencies.jar --name Ankitha --age 12 --subjects Science,Biology,Maths -p --isSpeciallyAbled --hostaddr domain_name:8080

Argument Validation:

Let's say, we want to validate the port field to be an integer. This can be done usingIParameterValidator interface. Create a class implementing this interface and override validate() method:

package org.example;

import com.beust.jcommander.IParameterValidator;
import com.beust.jcommander.ParameterException;

public class PortValidator implements IParameterValidator {

    @Override
    public void validate(String name, String value) throws ParameterException {
        try{
            Integer.parseInt(value.split(":")[1]);
        }
        catch (NumberFormatException ex) {
            throw new ParameterException("Parameter "+name+" should be integer");
        }
    }
}

In Argument class, include validateWith option:

    @Parameter(names = "--hostaddr", description = "hostname and port", converter = HostPortConverter.class, validateWith = {PortValidator.class})
    public HostPort hostPort;

If validation fails, we will get a ParameterException as configured in PortValidator class.

Note: We can provide multiple validators for any argument.

Main parameter:

So far, we have provided the parameters with the name option. But, we can have at most one parameter without a name. In Argument class:

    @Parameter(description = "main param")
    public String mainParam;

If we provide multiple fields without names option, it will throw ParameterException

Note: The main parameter need not be a required argument.
