API Code generation using Swagger in Springboot

API allows two systems to talk to each other. We can perform required CRUD operations using them in the target system i.e, Create, Read, Update or Delete of resources can be done using APIs. Today, we will look at how we can auto-generate the REST API code using swagger specification in a springboot application.

For this purpose, we need to include yaml file in our project code, based on which code will be generated in the mentioned location. To make use of this auto-generated code we need to implement generated interface and provide our own logic.

First of all, we need the below dependencies and configuration in our pom.xml file

<dependency>
   <groupId>io.springfox</groupId>
   <artifactId>springfox-swagger2</artifactId>
   <version>3.0.0</version>
</dependency>
<dependency>
   <groupId>javax.validation</groupId>
   <artifactId>validation-api</artifactId>
   <version>1.1.0.Final</version>
</dependency>

and below plugin with path to yaml file having API definition, location to store generated code.

<build>
   <plugins>
      <plugin>
         <groupId>io.swagger</groupId>
         <artifactId>swagger-codegen-maven-plugin</artifactId>
         <version>2.4.16</version>
         <executions>
            <execution>
               <goals>
                  <goal>generate</goal>
               </goals>
               <configuration>
                  <inputSpec>./api-def/apiDefinition.yaml</inputSpec>
                  <language>spring</language>
                  <configOptions>
                     <sourceFolder>swagger</sourceFolder>
                     <java8>true</java8>
                     <interfaceOnly>true</interfaceOnly>
                     <modelPackage>com.example.swaggercodegenerationoverview.model</modelPackage>
                     <apiPackage>com.example.swaggercodegenerationoverview.api</apiPackage>
                     <useBeanValidation>true</useBeanValidation>
                  </configOptions>
               </configuration>
            </execution>
         </executions>
      </plugin>
   </plugins>
</build>

As specified above, our API definitions are present in ./api-def/apiDefinition.yaml . Generated API code will be present in com.example.swaggercodegenerationoverview.api . If any model classes, they will be stored in com.example.swaggercodegenerationoverview.model

Under paths key, we need to specify all paths required to be generated. It need to have,

  1. API endpoint — path with any path params involved within braces
  2. HTTP method —the type of API i.e, get, put, post, delete, etc.
  3. summary — description of the API
  4. produces — format of the response
  5. tag — tag for the API
  6. responses — various response codes and their format
  7. parameters — parameters involved, query param or path param
  8. operationId — the name of the function to be generated

If any model classes are required, like in case of request payload or response. In that case, we need to provide definitions for them as well with all parameters involved.

$ref: '#/definitions/myElement' # means go to the root of the current document and then find elements definitions and myElement one after one.

Note: If the parameter value is predefined, we can made them as enums in yaml file
