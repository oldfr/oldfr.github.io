Performance Analysis and Improvements in JAVA

Efficient application uses resources to enhance its allocation with optimum operational cost-effectively. 

I recently worked on improving our application's performance. Since this was my first time conducting such an analysis, I read about different criteria that might contribute to memory leaks or long execution times.

Here, I will share my analysis of different criteria that could affect performance:

  1. Infrastructure—If you have a thread-heavy application, i.e., a CPU-bound application, you can increase the number of cores in the deployment environment.
  2. Parallelism techniques: Check the number of cores in the CPU and use it efficiently. Then, choose an approach where parallelism is configurable. For example, in Java, we can choose CompletableFuture, ExecutorService, or ForkJoinPool based on the requirement.
  3. Ideal heap size —  Suppose your service has allocated 12GB of memory. But, you set the minimum for heap size as 2GB and the maximum as 4 GB. You are not using the whole allocated memory. It will cap to the max size set. Choose ideal min and max values for the heap. 
  4. GC Options — The garbage collector plays an important role in removing unwanted objects from memory making space for new ones. Also, we should ensure this does not cause a delay in code execution. You can look here for all available GC options.
  5. Poor implementation — There is a chance that our code uses unnecessary iterations that cause delays in execution. Go with multiple levels of code reviews before moving it to production. Revisit the code for improvements in constant intervals.

Nowadays, many tools called profilers are available that collect and analyze multiple metrics helping to identify any memory leaks and performance bottlenecks. Some of the tools are JProfiler, Spring Profiler, etc,

Choosing Optimal GC Options:

Some applications must have low response times even under high load. To achieve this, along with other factors, we can use different GC tuning strategies that generally result in the best performance.

Certain tuning strategies are designed to yield the lowest average garbage collection times or minimize the frequency of full GCs. Those strategies can sometimes result in one or more very long periods of garbage collection (often several seconds long). To maintain our low response time, all long GC intervals must be avoided.


A key factor in maintaining brief response times is the proper selection and tuning of the JVM’s Garbage Collection (GC) algorithm.

  • -Xms, -Xmx: Places boundaries on the heap size to increase the predictability of garbage collection. -Xms sets the starting size to prevent pauses caused by heap expansion. -Xmx sets the maximum size the heap could grow.
  • -XX:+UseG1GC: Use the Garbage First (G1) Collector. Details can be found here
  • -XX:MaxGCPauseMillis: Sets a target for the maximum GC pause time. This is a soft goal, and the JVM will try to achieve it.
  • -XX:ParallelGCThreads: Sets the number of threads used during parallel phases of the garbage collectors. The default value varies with the platform on which the JVM is running.
  • -XX:ConcGCThreads: Number of threads concurrent garbage collectors will use. The default value varies with the platform on which the JVM is running.
  • -XX:InitiatingHeapOccupancyPercent: Percentage of the (entire) heap occupancy to start a concurrent GC cycle. GCs that trigger a concurrent GC cycle based on the occupancy of the entire heap and not just one of the generations, including G1, use this option. A value of 0 denotes ‘do constant GC cycles’. The default value is 45.


You can run java -help or java -X to see a list of available options and their descriptions.
