Singleton Design Pattern in Java

Software design patterns are reusable solutions that are not exactly the final product. These are the templates for software design. They are not a code that can be directly converted into the software. There are many design patterns present.

Design patterns can be broadly classified into 3 types:

  • Creational: designed for class instantiation. They can be either class-creation patterns or object-creational patterns.
  • Structural: designed with regard to a class’s structure and composition.
  • Behavioral: designed depending on how one class communicates with others.

Today, I will explain about Singleton design pattern which is a creational design pattern. Singleton means only one instance of the class can exist anytime.

To make a class Singleton:

  1. make all constructors private
  2. Have a static method that will return the object. Object creation can be lazy or eager instantiation.

Two forms of singleton design pattern

  • Early Instantiation: Creation of instance at load time.
  • Lazy Instantiation: Creation of instance when required. (better approach)

Lazy Instantiation:

public static SingletonPatternExample {
private static SingletonPatternExample singletonPatternObj = null;
private SingletonPatternExample() {
}
public static SingletonPatternExample getInstance() {
    if(singletonPatternObj == null) {
       singletonPatternObj = new SingletonPatternExample(); // lazy instantiation
    }
return singletonPatternObj;
}
}

Early Instantiation:

public static SingletonPatternExample {
private static SingletonPatternExample singletonPatternObj = new SingletonPatternExample(); // early instantiation
private SingletonPatternExample() {
}
public static SingletonPatternExample getInstance() {
    return singletonPatternObj;
}
}

Advantages:

  • Useful when only a single instance of the class should exist. e.g.: DB connection, sessionFactory in Hibernate, etc.
  • Saves memory as only one object will exist every time

Uses:

Thread pools, Configuration settings, logging

Serialization in singleton pattern

If the singleton class is Serializable, you can serialize the singleton instance. Once it is serialized, you can deserialize it but it will not return the singleton object.

To resolve this issue, you need to override the readResolve() method that enforces the singleton. It is called just after the object is deserialized. It returns the singleton object.
